# Phase 12 — OKF graph store (kill the Neo4j dependency)

> **Decision context:** `OKF-vs-Neo4j-report.md` (repo root) established that Warp Compass uses
> Neo4j only as a typed key-value+edge store — every real graph operation (completeness, chain
> validation, cross-persona verification, docgen) already runs in Python over an in-memory
> snapshot. This phase replaces Neo4j with an **OKF-style Markdown bundle** behind the existing
> `GraphStore` seam, deleting the DB server, the "start Neo4j" operator step, and the `neo4j`
> dependency entirely. ADR #28 records the decision.

---

## 1. Goals (from the owner)

1. **Remove Neo4j completely** — no driver, no Desktop app, no start step.
2. **Readable, LLM-friendly knowledge base** — every node is a Markdown file a human or an
   agent can open: *a few keywords as the primary identifier + a few sentences of description*,
   so an LLM can cheaply decide "use this node or create a new one".
3. **Two-way cross-links** — when A links to B, **both** files show the relationship
   (`[[node-id]]` wiki-links: outgoing under *Links*, incoming under *Backlinks*).
4. **Active-editor ingestion** — distilled, timestamped facts; never raw transcript copies;
   personal names abstracted into organizational roles.
5. **Architecture unchanged** — mobile users → PWA → Apps Script → Drive → laptop brain →
   briefs → Drive → PWA. The bus, the runner, the contracts, the round cycle: untouched.
6. **G-Drive story** — the graph itself now lives *inside* the Drive-synced engagement folder,
   so the knowledge base is auto-backed-up and readable from anywhere, with zero new setup.

## 2. What OKF gives us (and the profile we add)

OKF v0.1 (Google Cloud, 2026-06-12) = a directory of Markdown files with YAML frontmatter;
one file per concept; the file path is the concept's identity; links are ordinary Markdown
links; the only required field is `type`. It is a *format*, not a database — perfect here,
because all reasoning is already in-memory Python.

Vanilla OKF links can't carry per-edge provenance/status, which Warp Compass needs (handoff
verification lives on edges). So we define a small **producer profile**: relationships are
stored machine-readably in frontmatter (`edges:` list on the *from* node, each with type,
target and provenance), while the generated Markdown body renders them as human-readable
`[[wiki-links]]` in both directions.

## 3. Bundle layout

```
{GRAPH_ROOT}/                      # default: {BUS_ROOT}/graph  → synced by Drive for Desktop
  index.md                         # generated: bundle overview + per-type counts + links
  roles/
    index.md                       # generated: one line per node (id, title, keywords, status)
    role.sales-manager.md
  activities/
    act.check-stock.md
  systems/ · artifacts/ · events/ · approval-points/ · rules/ · problems/ · desires/ · kpis/
```

- **File path = identity**: `activities/act.check-stock.md` ⇔ node id `act.check-stock`.
  The filename carries the full slug id, so the mapping is unambiguous in both directions.
- `index.md` files are regenerated on `close()` (OKF's progressive-disclosure convention).

## 4. Node file anatomy

```markdown
---
type: Activity                     # OKF's one required field = our ontology type
id: act.check-stock
title: Check stock                 # canonical_name
keywords:                          # aliases — the "few keywords" primary identifier
  - stock check
  - verify availability
description: >-                    # the "few sentences": what it is + why it exists
  Verify stock availability for an incoming order before it is confirmed.
status: confirmed                  # derived: conflicting > confirmed > unverified > proposed
category_codes: ["02"]
key_attributes:
  exceptions: partial stock → split order
timestamp: 2026-07-01T09:12:00Z    # latest provenance ts (OKF reserved field)
provenance:                        # who said it, when, how sure — full history
  - said_by: participant.abc
    session_id: s_41
    confidence: 0.7
    status: confirmed
    ts: 2026-07-01T09:12:00Z
edges:                             # OUTGOING relationships (machine truth, with provenance)
  - type: HANDS_OFF_TO
    to: role.inventory-lead
    provenance: [ ... ]
---
# Check stock

> **Activity** `act.check-stock` · status **confirmed** · keywords: stock check, verify availability

Verify stock availability for an incoming order before it is confirmed.

## Facts
- 2026-07-01 — participant.abc (session s_41, confidence 0.7, confirmed)

## Links
- HANDS_OFF_TO → [[role.inventory-lead]] — Inventory Lead

## Backlinks
- [[role.sales-rep]] (Sales Rep) — PERFORMS

<!-- auto-generated by warp-compass-brain; edit via the pipeline, not by hand -->
```

- **Frontmatter is the machine truth** — it round-trips `NodeCard`/`Edge` exactly (pydantic
  validated on load). `keywords` ⇔ `aliases`; `title` ⇔ `canonical_name`.
- **The body is a generated view**, rewritten on every persist from the in-memory graph. That
  is what makes the two-way `[[link]]` invariant *structural*: `add_edge(A→B)` rewrites both
  A (Links) and B (Backlinks); nothing can drift.
- Timestamped facts = the `## Facts` list rendered from provenance (said-by + session + ts).

## 5. `OkfGraphStore` semantics (drop-in for `Neo4jGraphStore`)

| Operation | Implementation |
|---|---|
| `connect()` | Walk the bundle, parse frontmatter (PyYAML) → in-memory `{id: NodeCard}` + edge list. Malformed files are skipped with a stderr warning (tolerant-reads, like `FolderBus`). |
| reads (`get_node`, `find_by_alias`, `neighbors`, `nodes_by_type`, `edges`) | Pure in-memory; return deep copies (same isolation contract as before). |
| `upsert_node` | Update memory, atomically rewrite that node's file (tmp + `os.replace`), plus any files whose Backlinks mention it (title may have changed). |
| `add_edge` | Idempotent merge on `(type, from, to)` — re-adding overwrites provenance (Neo4j `MERGE … SET` semantics). Rewrites **both** endpoint files. |
| `set_status` | Sets the last provenance status + refreshes the derived `status` field (mirrors `FakeGraphStore`; effective status was always derived from provenance — ADR #24b). |
| `close()` | Regenerate `index.md` files if anything changed. |
| `query()` | **Removed from the `GraphStore` ABC** — it was Cypher-specific and only ever used by tests. |

Scale check: a full engagement is O(10²–10³) nodes → full load is milliseconds; a round's
writes are hundreds of small atomic file writes. No server, no locks (single-writer brain, per
the existing operating model).

## 6. How Google Drive works now

**Nothing about the sync architecture changes.** The bus (`{BUS_ROOT}/participants/…`) still
carries Answer Logs in and briefs out via Apps Script + Drive for Desktop (P11). The only new
thing: the graph bundle defaults to **`{BUS_ROOT}/graph/`**, i.e. *inside the same Drive-synced
folder* — so the knowledge base is continuously backed up to Drive and browsable from any
device, with **zero** new configuration. `GRAPH_ROOT` in `brain/.env` overrides the location
(e.g. to keep the graph local-only).

Notes:
- The brain remains the **only writer** of `graph/` (mobile users never touch it) — no
  concurrent-write conflicts for Drive to mangle.
- Writes are atomic (`tmp` + `os.replace`), so Drive never uploads a half-written node.
- The vector index (`brain/_state/vectors.sqlite`) stays **local** — it's a rebuildable cache,
  and syncing a live SQLite file over Drive is exactly the kind of thing that corrupts.

## 7. Migration & rebuild paths

1. **From a live Neo4j** (one-off): `scripts/migrate_neo4j_to_okf.py` — a standalone script that
   is *not* part of the package; run `uv run --with neo4j python ..\scripts\migrate_neo4j_to_okf.py`
   from `brain/`. It bolt-reads every node/edge and writes the bundle via `OkfGraphStore`.
   After migration, Neo4j Desktop can be uninstalled.
2. **From the Answer Logs** (always available): the graph is re-derivable (ADR #4). Clear each
   `profile.json`'s `ingested_logs` and `run-round` re-ingests everything (costs LLM calls).

## 8. Also in this phase — interviewer refocus (owner feedback)

Field tests showed the bot "picking from here and there" and leading with *"what's the most
difficult part of your job?"*. The end goal is the **SOP** — a complete 0→100 picture of each
role — not a pain-point survey. Changes:

- **`SYSTEM_PROMPT`** (pwa/src/runner/prompts.ts): explicit mission (complete end-to-end SOP of
  this role), ground-up chronological method (skeleton of the day first, then depth per
  activity, following "what happens next?"), and a hard rule to never lead with
  difficulty/frustration questions — problems are captured when they surface naturally.
- **`COLD_START_OPENERS`** (prompts.ts + planner.py, mirrored constants): rewritten as a
  chronological walk (role → first thing that lands → step by step → inputs/tools per step →
  who picks it up next). The frustration opener is gone.
- **Extractor prompt** (extractor.py): active-editor rules — distill into factual sentences
  (never copy conversational text), `canonical_name` + keyword aliases as the identifier,
  2–3 sentence descriptions (what it is + why it exists), and **personal names are abstracted
  to organizational roles** (a name like "John" must become the role he plays, e.g.
  "Accounts Approver" — people change, roles persist).
- Thread ranking (threads.py) already prioritizes structural SOP fields (handoff/trigger/output
  top the field impacts; `exceptions` lowest) — unchanged.
